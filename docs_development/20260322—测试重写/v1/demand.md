# 需求文档：测试系统重写

> 创建时间：2026-03-22
> 版本：v1

---

## 1. 需求背景

当前项目的测试存在以下问题：

### 1.1 测试质量问题

| 问题类型 | 示例文件 | 具体表现 |
|---------|---------|---------|
| **虚假测试** | `loop.test.ts` | 只验证对象创建（`expect(loop).toBeDefined()`），不验证任何实际行为 |
| **断言不足** | `loop.test.ts` | 大量测试只检查构造函数接受参数，不验证配置是否正确应用 |
| **类型滥用** | `read.test.ts` | 频繁使用 `as any` 类型断言，绕过类型检查 |
| **Mock 过度** | 多个测试文件 | 为避免测试而大量 Mock，导致测试与实际代码脱节 |
| **覆盖不足** | 大部分测试 | 只测试 happy path，缺少边界条件和错误处理测试 |
| **隔离不足** | 集成测试 | 集成测试与单元测试混在一起，Mock 不完整 |

### 1.2 当前测试结构

```
tests/
├── src/                    # 单元测试
│   ├── agents/             # Agent 核心 (loop, supervisor, prompt...)
│   ├── tools/              # 工具 (file, bash, web...)
│   ├── api/                # API 路由
│   └── ...
├── integration/            # 集成测试
│   ├── agent.test.ts
│   ├── tool-loop.test.ts
│   └── ...
└── e2e/                    # E2E 测试
    ├── session-e2e.test.ts
    └── ...
```

## 2. 详细需求描述

### 2.1 核心目标

按照 **TDD 规范** 重写所有测试，实现：
- **80%+ 测试覆盖率**
- **真实行为验证**（不只是 `toBeDefined()`）
- **完整的边界条件和错误处理测试**
- **清晰的测试分层**（单元/集成/E2E）

### 2.2 测试分层重构

#### 单元测试 (src/)
- 每个测试文件对应一个源代码文件
- 使用真实的 Mock 而非 `as any`
- 测试函数/类的核心逻辑和边界条件

#### 集成测试 (integration/)
- 测试多个组件协作
- 使用真实的存储（SQLite 内存数据库）
- 测试 API 端点到数据库的完整链路

#### E2E 测试 (e2e/)
- 测试完整用户流程
- 使用 Playwright 进行浏览器自动化
- 覆盖关键用户场景

### 2.3 测试质量标准

| 标准 | 描述 |
|-----|------|
| **可读性** | 测试名称清晰描述测试场景 |
| **独立性** | 每个测试不依赖其他测试的状态 |
| **可重复性** | 相同结果可在任何环境复现 |
| **快速执行** | 单元测试 < 100ms/个 |
| **真实断言** | 验证实际行为，不只是对象创建 |

### 2.4 Mock 策略

| 场景 | Mock 对象 | 替代方案 |
|-----|----------|---------|
| LLM 调用 | Mock LLM client | 使用 Fake LLM (预设响应) |
| 文件系统 | Mock fs | 使用临时目录 + 真实文件操作 |
| 外部 API | Mock fetch | 使用 MSW 或 nock |
| 时间相关 | Mock Date | 使用假时钟 (sinon.fakeTimers) |

## 3. 期望结果

### 3.1 覆盖率目标

```
Module          | Coverage | Notes
----------------|----------|------
agents/         | 85%+     | 核心逻辑全部覆盖
tools/          | 80%+     | 重点工具完整覆盖
api/routes/     | 90%+     | 所有端点及错误码
memory/         | 80%+     | 存储操作覆盖
session/        | 85%+     |
```

### 3.2 交付物

1. **重构后的测试文件** - 按照 TDD 规范重写
2. **测试覆盖率报告** - 使用 Vitest coverage
3. **测试文档** - 关键测试场景说明

---

## 4. 约束条件

- 必须使用 Vitest 作为测试框架
- 必须使用项目已有的 Mock 设施（`tests/mocks/`）
- 不能破坏现有的 CI/CD 测试流程
- 重写过程中保持 API 兼容性

---

## 5. 优先级

| 优先级 | 模块 | 原因 |
|-------|-----|------|
| P0 | `agents/loop.ts` | 核心循环，测试最薄弱 |
| P0 | `tools/file/` | 文件操作，高频使用 |
| P1 | `session/` | 会话管理，状态复杂 |
| P1 | `memory/` | 记忆系统，关键功能 |
| P2 | `api/routes/` | API 层，相对完善 |
